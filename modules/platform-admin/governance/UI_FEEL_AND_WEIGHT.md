## 1) Purpose

This document defines **calm vs tension** in operational terms and establishes rules for controlling visual weight **WITHOUT clutter**.

**This is a governance artifact, NOT a design deliverable.**

Contains **NO** visual specifications or implementation details.

**Evidence**: Gate 29.5 requirements

---

## 2) Calm vs Tension (Operational Definition)

### Calm

**Operational characteristics**:

- Predictable behavior
- Stable layouts
- Controlled contrast
- Minimal visual noise
- Functional-only motion
- Clear hierarchy
- Consistent patterns

**User experience**:

- Feels supportive
- Reduces cognitive load
- Enables focus
- Supports long sessions

**Evidence**: Brand & UI Constitution Section 1, Bassan_OS_Platform_Vision_Document.MD Section 4

---

### Tension

**Operational characteristics**:

- Unpredictable behavior
- Changing layouts
- High contrast
- Visual clutter
- Decorative motion
- Unclear hierarchy
- Inconsistent patterns

**User experience**:

- Feels demanding
- Increases cognitive load
- Reduces focus
- Causes fatigue

**Evidence**: UI_Density_Policy.MD Section 2

---

## 3) Visual Weight Control (WITHOUT Clutter)

### Definition

**Visual weight** = Perceived calmness vs tension.

**NOT**: Number of elements.

**Evidence**: Gate 29.5 terminology lock

---

### Control Mechanisms

Visual weight is controlled through:

- **Spacing**: Adequate breathing room between elements
- **Contrast**: Controlled contrast levels
- **Hierarchy**: Clear visual hierarchy
- **Consistency**: Predictable patterns
- **Motion**: Functional-only motion (150–250ms)
- **Color**: Soft backgrounds, controlled accents

**NOT through**:

- Removing necessary elements
- Hiding functionality
- Reducing information density arbitrarily

**Evidence**: Brand & UI Constitution Sections 3, 5

---

## 4) Rules for Adding Elements

### When to Add Elements

**MUST add** when:

- Element serves clear functional purpose
- Element reduces cognitive load
- Element improves orientation
- Element prevents errors
- Element supports long-session work

**Evidence**: Brand & UI Constitution Section 16

---

### When NOT to Add Elements

**MUST NOT add** when:

- Element is decorative only
- Element duplicates existing functionality
- Element increases visual noise
- Element adds cognitive load
- Element serves marketing purposes

**Evidence**: Brand & UI Constitution Section 16

---

## 5) Rules for Removing Elements

### When to Remove Elements

**MUST remove** when:

- Element is decorative only
- Element duplicates existing functionality
- Element increases visual noise without benefit
- Element adds cognitive load without benefit
- Element serves no functional purpose

**Evidence**: Brand & UI Constitution Section 17

---

### When NOT to Remove Elements

**MUST NOT remove** when:

- Element serves functional purpose
- Element reduces cognitive load
- Element improves orientation
- Element prevents errors
- Element supports long-session work

**Evidence**: Brand & UI Constitution Section 16

---

## 6) UI Behavior Under Long Usage

### After 4–10 Hour Sessions

UI must maintain:

- Visual calmness (no fatigue from high contrast)
- Cognitive clarity (no overload from clutter)
- Spatial orientation (no disorientation from layout changes)
- Behavioral predictability (no anxiety from unpredictable behavior)

**Evidence**: UI_Density_Policy.MD Section 11, Brand & UI Constitution Section 20

---

### Fatigue Prevention

**MUST avoid**:

- High visual contrast causing eye strain
- Visual clutter causing cognitive overload
- Sudden layout changes causing disorientation
- Unpredictable behavior causing anxiety

**Evidence**: UI_Density_Policy.MD Section 2

---

## 7) UI Behavior Under Operational Pressure

### When Users Are Under Pressure

UI must provide:

- **Stability**: No sudden changes
- **Clarity**: Clear states and actions
- **Predictability**: Consistent behavior
- **Support**: Helpful feedback without nagging

**Evidence**: Brand & UI Constitution Section 14

---

### Forbidden Under Pressure

**MUST NOT**:

- Add attention-grabbing effects
- Change layouts suddenly
- Introduce decorative animations
- Show unclear states
- Nag with repeated notifications

**Evidence**: Brand & UI Constitution Sections 5, 12

---

## 8) Explicitly Forbidden Visual Behaviors

### Impressive UI

**Forbidden**: UI designed to impress or wow users.

**Rationale**: The UI exists to serve people, not to impress them.

**Evidence**: Brand & UI Constitution Section 1

---

### Over-Visualization

**Forbidden**: Excessive use of charts, graphs, and visual representations.

**Rationale**: Over-visualization increases cognitive load in operational contexts.

**Evidence**: Bassan_OS_Platform_Vision_Document.MD Section 5

---

### Feature-Showcasing Layouts

**Forbidden**: Layouts designed to display all available features prominently.

**Rationale**: Feature showcasing creates visual clutter and overwhelms users.

**Evidence**: Brand & UI Constitution Section 17

---

### Visual Noise Creep

**Forbidden**: Gradual accumulation of decorative elements over time.

**Rationale**: Visual noise creep destroys calm and increases fatigue.

**Evidence**: Brand & UI Constitution Section 17

---

## 9) Visual Weight Philosophy Summary

### Core Principle

> Visual weight must be calm, balanced, low-noise, structured, and functional.

**Evidence**: Bassan_OS_Platform_Vision_Document.MD Section 7

---

### Control Strategy

Visual weight is controlled through:

- Adequate spacing (not removing elements)
- Controlled contrast (not reducing readability)
- Clear hierarchy (not hiding functionality)
- Consistent patterns (not changing layouts)
- Functional motion (not decorative effects)

**Evidence**: Brand & UI Constitution Sections 3, 5
